# simple-asana-ui

Asana dashboard hosted on Cloudflare Pages: a React/Vite/Tailwind front-end (source in `ui/web/`, built to the static `public/` output directory) plus Pages Functions in `functions/` implementing an Asana OAuth 2.0 authorization-code flow and a same-origin API proxy. **`BUILD_SPEC.md` is the authoritative spec тАФ read it before implementing or changing any auth/proxy behavior, and do not invent scope beyond it.**

## Commands

- `npm run dev` тАФ local dev server: `wrangler pages dev` proxying to the Vite dev server, so functions + hot-reloading UI run together
- `npm run build` тАФ `vite build` (front-end, outputs to `public/`) then `wrangler pages functions build` (functions validation only; Cloudflare Pages builds functions itself on deploy)
- `npm run lint` тАФ ESLint (flat config in `eslint.config.js`)
- `npm run format` / `npm run format:check` тАФ Prettier write / check

Run `npm run lint`, `npm run format:check`, and `npm run build` before committing.

## Verification

Do not verify changes by launching or driving the app in a browser (no headless Chrome, Playwright, screenshots, or interactive browser sessions). Verify via `npm run lint`, `npm run format:check`, `npm run build`, and careful code review against the spec/design only.

## Layout

- `ui/web/` тАФ React/Vite/Tailwind source for the front-end. Component-based; wire it to the fetch/401/logout contract in BUILD_SPEC.md. Responsive: the same build serves desktop and phone тАФ layout switches at Tailwind's `md` breakpoint (768px) via responsive classes plus the `useIsMobile` hook (`ui/web/src/lib/useIsMobile.js`) for structural differences (stacked task rows, full-screen task detail, collapsible filters). There is no separate mobile build.
- `public/` тАФ build output directory (gitignored, generated by `vite build`). Do not hand-edit.
- `functions/auth/` тАФ `login.js`, `callback.js`, `logout.js` (OAuth flow)
- `functions/api/asana/[[path]].js` тАФ catch-all proxy to `https://app.asana.com/api/1.0/`
- `functions/_lib/cookies.js` тАФ cookie parse/serialize + token refresh helpers

## Hard constraints (from the spec)

- Front-end is a React (Vite + Tailwind) build that compiles to a static `public/` directory тАФ no server-side rendering, no Node server at runtime. No standalone Workers, no `wrangler.toml`, no KV/D1/database. Wrangler is a dev/CI tool only; deploys happen automatically via Cloudflare Pages git integration.
- The Asana client secret and tokens must never reach the browser: secrets in Pages env vars (`ASANA_CLIENT_ID`, `ASANA_CLIENT_SECRET`, `ASANA_REDIRECT_URI`, `ASANA_ALLOWED_WORKSPACE_GID`), tokens in `HttpOnly; Secure; SameSite=Lax` cookies (`asana_at`, `asana_rt`, `oauth_state`).
- The front-end only calls same-origin `/api/asana/*`, never `app.asana.com` directly.
- Workspace-membership gate is enforced server-side in the callback.
- Never log token values. Proxy returns 401 (not 500) when unauthenticated.
- Don't rely on training-data knowledge of the Asana API тАФ fetch current docs first, starting at `https://developers.asana.com/llms.txt`.
